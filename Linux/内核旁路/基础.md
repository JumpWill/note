# 内核旁路

## 概念

内核旁路 = 应用/驱动直接跟硬件（NIC/NVMe）打交道，跳过内核协议栈的 copy + 调度 + 锁，把延迟从 ms/μs 压到 μs/ns 级。

## 为什么内核是瓶颈

正常收一个 TCP 包的路径：

```text
NIC → DMA → NIC 驱动 → 内核协议栈 (skb → IP → TCP) → copy_to_user → 应用 recv()
```

内核在这里做的：中断、软中断、协议解析、锁、memcpy。

四个主要代价：

* 两次 memcpy（NIC→skb，内核→用户）
* 中断 + 软中断（上下文切换 + 调度抖动）
* 内核锁竞争（多核下 skb/套接字共享）
* 协议栈逻辑（TCP 状态机、重排、拥塞）

## 主流技术栈

### DPDK

Intel 开源，用户态 Polling Mode Driver (PMD)。NIC 收到包 DMA 到用户态 hugepage，零拷贝、零中断。`while(1) rte_eth_rx_burst()` 轮询，CPU 绑核独占。

典型延迟 ~1μs 收包（内核栈 10–50μs）。

硬件要求：

* NIC：Intel i40e/ice/e1000、Mellanox mlx4/mlx5、Chelsio、Netronome NFP；普通 Realtek 板载不在 PMD 支持列表
* 主板：多核 + NUMA 友好，绑核需要 `isolcpus` 隔离核
* 内存：1GB / 2MB hugepage，BIOS 关节能模式 / C-State
* 驱动：用 DPDK PMD 就不能让内核 igb/uio 同时接管同一端口（绑定到 `vfio-pci` 或 `igb_uio`）
* OS：Linux 主流，FreeBSD 也有 port，Win 无
* 容量预估：单核 ~10–20 Mpps，10M pps 至少独占 1 核

### RDMA / RoCE / iWARP / InfiniBand

Remote DMA，A 机用户态直接读 B 机内存，远端 CPU 零参与。

两种底层：

* InfiniBand (IB)：专用 HCA + 交换机，< 1μs
* RoCEv2：以太网跑 RDMA，便宜，需无损网络（PFC + ECN）

BeeGFS / Lustre / 部分 KV 存储用它做传输层。

硬件要求：

* HCA：Mellanox ConnectX-4 起，建议 ConnectX-5/6/7（支持 RoCEv2 + GPUDirect）
* 交换：IB 用 Mellanox Quantum/Switch-X 系列；RoCE 用支持 PFC/ECN 的 ToR（Arista 7060、Cisco Nexus 9336c）
* 线缆：IB 用 QSFP28/AOC；RoCE 用 25G/100G DAC 或光模块，禁用 1G
* 网络：RoCE 必须无损，PFC 流控 + ECN 标记 + DCB 队列，带宽抖动即丢包
* OS：Linux rdma-core 驱动，Win 有 WinOF2

### io_uring

Linux 5.1+ 引入，不是完全 bypass，但大幅削减 syscall + 上下文切换。

* 传统：`recv()` 一次往返一次 syscall
* io_uring：用户态 SQ 塞 N 个请求，内核消费，CQ 批量取结果，POLL 模式还能免唤醒
* 网络 + 存储都能用

硬件要求：

* 无专用 NIC，普通网卡就能跑
* 推荐多核 + NVMe SSD（io_uring + NVMe 是当前 Linux 文件 I/O 天花板）
* 内核：5.6+（网络 send/recv），5.11+（POLL 模式稳定），5.19+（zero-copy send）
* OpenEBS Mayastor = io_uring + NVMe 走到底，K8s 里 PV 延迟天花板

### XDP / eBPF

eBPF 程序挂到 NIC 驱动层，包进 DMA 后、进协议栈前截住。可：DROP / TX / 改包 / REDIRECT 到用户态（AF_XDP）。

用在 DDoS 清洗（Cilium / Cloudflare）、LB。

硬件要求：

* 驱动支持 XDP native 模式：i40e/ice/mlx5/tun/veth/loopback 等；部分驱动只支持 generic XDP（性能差）
* 推荐 NIC：Intel XXV710 / Mellanox ConnectX-5+，AF_XDP zero-copy 还要驱动支持
* AF_XDP 零拷贝额外要求：UMEM hugepage + 网卡 RSS 对齐队列
* 普通 Realtek / r8169 板载只能跑 generic XDP，延迟不如 native

### SPDK

类比 DPDK 但针对 NVMe：用户态 PMD + poll mode，跳过内核块层 + blk-mq。延迟：内核块层加 ~30μs，SPDK 回到 NVMe 物理 ~10μs。

硬件要求：

* NVMe：NVMe-oF / PCIe NVMe SSD（Intel Optane、Samsung PM9A3、Kioxia CM7）；SATA SSD 收益有限
* CPU：同 DPDK，绑核独占；现代 CPU 配 NVMe 才能跑满队列深度
* 驱动：`vfio-pci` 把 NVMe 控制器绑到用户态，不能用内核 `nvme` 同时接管
* 文件系统：需 SPFS / BlobFS 等用户态 FS，或自建（Ceph OSD 改 SPDK 是一大工程）
* 主板：PCIe 3.0 x4 起步，U.2 / AIC 形态更稳；M.2 在多盘场景散热差

## 技术栈对照

* DPDK：旁路协议栈 + 中断，~1μs，需 PMD-supported NIC + 1G hugepage，HFT / NFV / LB
* RDMA (IB/RoCE)：旁路内核 + 远端 CPU，<1μs，需 ConnectX-5+ + 无损网络，HPC / 分布式存储 / GPU 集群
* SPDK：旁路内核块层，~10μs，需 NVMe SSD + vfio-pci，本地盘 / Ceph OSD 重构
* io_uring：减少 syscall，介于内核/旁路之间，需普通 NVMe + Linux 5.11+，K8s PV / 通用高并发
* XDP：协议栈入口前截包，近 DPDK，AF_XDP 持平，需 Native-XDP NIC，Cilium / DDoS / 流量染色
* 内核默认栈：啥都不绕，10–50μs，任意硬件，通用业务

## 实现示例

### DPDK 收包

硬件前提：Intel XXV710（i40e 驱动）、绑定到 `vfio-pci`、系统配 1G hugepage、CPU 核 4 隔离给 DPDK。

```c
/* SPDX-License-Identifier: BSD-3-Clause */
#include <rte_eal.h>
#include <rte_ethdev.h>
#include <rte_mbuf.h>

#define PORT_ID   0
#define QUEUE_ID  0
#define BURST     32

int main(int argc, char *argv[]) {
    /* 1. 初始化 EAL（解析 -l core list、--file-prefix 等） */
    if (rte_eal_init(argc, argv) < 0) rte_exit(EXIT_FAILURE, "EAL init failed\n");

    /* 2. 端口配置：1 个 RX 队列 + 1 个 TX 队列 */
    struct rte_eth_conf port_conf = {0};
    if (rte_eth_dev_configure(PORT_ID, 1, 1, &port_conf) < 0)
        rte_exit(EXIT_FAILURE, "configure failed\n");

    /* 3. 分配 mbuf 池（从 hugepage 拿） */
    struct rte_mempool *mbuf_pool = rte_pktmbuf_pool_create(
        "MBUF_POOL", 8192, 256, 0, RTE_MBUF_DEFAULT_BUF_SIZE,
        rte_socket_id());
    if (!mbuf_pool) rte_exit(EXIT_FAILURE, "mbuf pool failed\n");

    /* 4. 配 RX / TX 队列 */
    rte_eth_rx_queue_setup(PORT_ID, QUEUE_ID, 1024,
                           rte_eth_dev_socket_id(PORT_ID), NULL, mbuf_pool);
    rte_eth_tx_queue_setup(PORT_ID, QUEUE_ID, 1024,
                           rte_eth_dev_socket_id(PORT_ID), NULL);
    rte_eth_dev_start(PORT_ID);

    /* 5. 死循环轮询——一个核就耗在这里 */
    struct rte_mbuf *bufs[BURST];
    while (1) {
        uint16_t nb_rx = rte_eth_rx_burst(PORT_ID, QUEUE_ID, bufs, BURST);
        for (uint16_t i = 0; i < nb_rx; i++) {
            /* 在这里做协议解析 / 改包 / 转发，全部用户态，无任何 syscall */
            rte_pktmbuf_free(bufs[i]);
        }
    }
    return 0;
}
```

运行：

```bash
# 启动
sudo dpdk-devbind.py --bind=vfio-pci 0000:01:00.0   # 把 NIC 给 vfio
echo 1024 > /proc/sys/vm/nr_hugepages                # 1G 大页
./app -l 4 -n 1 -- --no-mlockall                     # 绑到核 4
```

### XDP 早截包

硬件前提：i40e / mlx5 等 native-XDP 驱动；AF_XDP zero-copy 还需 ConnectX-5+。

```c
/* xdp_drop_tcp_syn.c */
#include <linux/bpf.h>
#include <linux/if_ether.h>
#include <linux/ip.h>
#include <linux/tcp.h>
#include <bpf/bpf_helpers.h>

/* 简单粗暴：所有 TCP 包直接 DROP，验证 XDP hook 通了 */
SEC("xdp")
int xdp_drop_all(struct xdp_md *ctx) {
    void *data     = (void *)(long)ctx->data;
    void *data_end = (void *)(long)ctx->data_end;

    struct ethhdr *eth = data;
    if ((void *)(eth + 1) > data_end) return XDP_PASS;

    if (eth->h_proto != __constant_htons(ETH_P_IP))
        return XDP_PASS;

    struct iphdr *ip = (void *)(eth + 1);
    if ((void *)(ip + 1) > data_end) return XDP_PASS;

    if (ip->protocol == IPPROTO_TCP)
        return XDP_DROP;   /* TCP 包在协议栈之前就丢掉 */

    return XDP_PASS;        /* 其他放行给内核正常处理 */
}

char _license[] SEC("license") = "GPL";
```

加载：

```bash
# 编译
clang -O2 -target bpf -c xdp_drop_tcp_syn.c -o xdp_drop_tcp_syn.o

# 挂到网口（native 模式，要求驱动支持）
ip link set dev eth0 xdp obj xdp_drop_tcp_syn.o sec xdp verbose

# 卸载
ip link set dev eth0 xdp off
```

## tick 接收场景对比

行情 tick 通常是组播 UDP，单条 50–200B，要求低延迟 + 高吞吐 + 低抖动。交换机侧普遍千兆 / 万兆 NIC，单核独占假设。

* 内核默认栈（UDP + recvfrom）
  * p50 ~20–30μs，p99 ~80–150μs，p999 抖动能到 500μs+
  * 单核吞吐 ~30–60 万 msg/s，CPU 利用率 60–80%
  * 瓶颈：软中断 + 协议栈 + skb 分配 + memcpy
* io_uring + XDP（recvmsg 批量）
  * p50 ~5–10μs，p99 ~20–40μs，p999 ~80μs
  * 单核吞吐 ~150–300 万 msg/s，CPU 50–70%
  * 仍走内核 UDP 协议栈（无连接组播跳过 TCP 状态机），主要省 syscall
* XDP native + kernel 协议栈（旁路 DROP/统计，packet 仍交内核）
  * p50 ~2–5μs，p99 ~8–15μs，p999 ~30μs
  * 单核吞吐 ~500–1000 万 msg/s（pps 极限），CPU 40–60%
  * 适合 tick 解析 / 过滤 / 染色后再交内核
* AF_XDP zero-copy（XDP 把 packet 直接送用户态 ring）
  * p50 ~1–2μs，p99 ~3–5μs，p999 ~10μs
  * 单核吞吐 ~1000–2000 万 msg/s，CPU 60–80%（独占核）
  * 解析逻辑全部用户态，组播接收几乎无拷贝
* DPDK（用户态 PMD 死循环 poll）
  * p50 ~0.5–1μs，p99 ~1–3μs，p999 ~5μs
  * 单核吞吐 ~2000–3000 万 msg/s，CPU 90–100%（独占核）
  * 抖动最小，标准差通常 < 0.5μs
* RDMA 行情源（交易所 / 一级 vendor 提供）
  * p50 ~0.3–0.5μs，p99 < 1μs，p999 ~2μs
  * 吞吐看硬件，ConnectX-7 单端口 200GbE 能扛 1 亿+ msg/s
  * 不走网络协议栈，硬件直接 DMA 到用户态 mbuf

数数量级感：

* 内核栈到 DPDK：延迟降 20–50 倍，吞吐涨 30–50 倍
* io_uring 是性价比甜点：相比内核栈延迟砍一半、吞吐翻 5 倍，硬件零改造
* XDP/AF_XDP 是中间过渡：保留内核能力 + 拿到 DPDK 70–90% 性能
* DPDK/RDMA 是"花钱换 μs"的终局，只有 SLA 硬约束（μs 级）才上

## eBPF/XDP vs io_uring

两个最常被一起拎出来问，定位其实完全不同。

旁路点：

* eBPF/XDP：包进 DMA 后、进协议栈**之前**就截住。完全跳过协议栈，但 packet 仍交给内核处理（DROP/TX/REDIRECT 除外）
* io_uring：包走完正常协议栈，只是用 SQ/CQ ring 把 syscall 批量掉，本质不旁路协议栈

性能（tick / UDP 接收场景）：

* eBPF/XDP（native）：p50 ~2–5μs，单核 ~500–1000 万 pps
* io_uring：p50 ~5–10μs，单核 ~150–300 万 msg/s
* 极限差异：XDP 在 native 模式下大概比 io_uring 快 2–5 倍

硬件要求：

* eBPF/XDP：要 native XDP 驱动（i40e/ice/mlx5/tun/veth），普通板载 NIC 只能跑 generic 模式（性能差）
* io_uring：跟 NIC 解耦，virtio-net / ENA 都能跑

云支持：

* eBPF/XDP：云上基本只剩 generic 模式，吞吐优势被吃光，只适合做流量染色 / 简单统计
* io_uring：完整支持，是云上唯一能拿到真收益的方案

编程模型：

* eBPF/XDP：C/Rust 写 eBPF 程序，过 verifier（5K instruction 上限、loop 受限），挂载点单一（驱动层）
* io_uring：liburing / iouring-go / boost::asio 多种封装，几乎所有语言都有支持

适用场景：

* eBPF/XDP：DDoS 清洗（Cilium / Cloudflare）、LB、流量染色、协议栈前的硬过滤
* io_uring：通用文件 / 网络 / 定时器 I/O，K8s 存储（Mayastor）、KV 存储、网络服务

可以同时用：

* XDP 在驱动层做第一道过滤 / 统计，命中要求的 packet 交给内核协议栈
* 应用层用 io_uring 从 socket 批量收，避开 syscall 开销
* 组合后 p50 ~3–5μs（接近 XDP 极限），p99 ~10–20μs（比纯 io_uring 好）
* 前提：native XDP 驱动 + 业务真的有"协议栈前丢弃"的需求

简单记忆法：

* "在协议栈前面做事" → XDP
* "让协议栈跑得更快" → io_uring

## 选型决策

优先上 io_uring 的信号：

* 业务卡在 syscall 占比、QPS 几千到几万、需要保留内核协议栈语义
* 不想换 NIC、不愿绑核、想保留 `tcpdump` `conntrack`

上 DPDK / RDMA 的信号：

* 延迟指标有硬性 SLA（μs 级），且能测出确实是 syscall + 协议栈在拖
* 有专用 NIC（XXV710 / ConnectX-5+）且机房能配大页 / 隔离核
* 团队里有人能长期维护底层 + 监控/调优

XDP 的信号：

* 想做在协议栈前面的一道前置处理（DDoS / 流量染色 / LB）
* 不愿意跳出内核、但又要接近 DPDK 的早处理点

不上：

* 通用业务、几千 QPS、延迟容忍 ms 级——内核的零拷贝 + 多路复用 + conntrack 早就够用，强行上 DPDK 只换运维负担
