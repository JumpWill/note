## 生态全景

eBPF 生态分四层，从底到顶：

```text
① 内核 / 基础设施   BPF subsystem + BTF + verifier + JIT + helper 函数
② 编译 / 加载器     clang/LLVM BPF target, libbpf, bpftool
③ 语言库            C libbpf / Rust aya / Go cilium-ebpf / Python BCC ...
④ 高层产品          Cilium, Falco, Pixie, Tetragon, Katran ...
```

---

## ① 内核基础设施

Linux 5.x 逐步完善，2026 现状：

* **BPF 子系统**：Alexei Starovoitov（Meta）+ Daniel Borkmann（Isovalent）+ Andrii Nakryiko（Meta）维护，5.18 起 Andrii 主线
* **BTF**（BPF Type Format）：内核自描述类型信息，是 CO-RE 的基础。`/sys/kernel/btf/vmlinux` 是只读 BTF
* **verifier**：保证 BPF 程序安全，是大爹也是性能瓶颈
* **JIT**：x86_64 / arm64 / riscv64 / powerpc 都支持
* **helper 函数**：内核暴露给 BPF 调用的白名单函数，约 200+ 个，持续增加
* **ringbuf**（5.8+）：替代 perf_event_array 的事件通道
* **bpf link**（5.8+）：内核持引用计数，是推荐的 attach 接口
* **fentry/fexit**（5.5+）：比 kprobe 更快更安全的现代 trace
* **sched_ext**（6.12+）：让 BPF 程序当调度器

发行版支持：Ubuntu 22.04+ / Debian 12+ / RHEL 9+ / Amazon Linux 2023+ 都开 BTF、kernel 6.x。容器镜像里要确认 `kernel.unprivileged_bpf_disabled=0` 或有 CAP_BPF。

## ② 编译 / 加载器

### clang / LLVM

* BPF target 是 `llvm/lib/Target/BPF/`
* 编译命令 `clang -target bpf -O2 -g -c app.bpf.c -o app.bpf.o`
* 必须开 BTF：`clang -target bpf -g ...` 会带 `-mattr=+btf`，CO-RE 重定位信息靠它

### bpftool

* 内核源码 `tools/bpf/bpftool/`，各发行版打包
* 子命令：`prog show / map show / prog dump xlated / prog profile id N / net attach xdp / gen skeleton`
* **gen skeleton** 是 libbpf 工作流的核心：从 .bpf.o 生成 `app.skel.h`

### libbpf

* 官方 C 库，包名 `libbpf-dev` / `libbpf1`
* 提供 ELF 解析、CO-RE 重定位、bpf() syscall 链、ring buffer poll API
* **libbpf-bootstrap**：Andrii 写的脚手架，开箱即用的 BPF 项目模板

## ③ 语言库

| 语言 | 主流库 | 维护方 | 特点 |
| --- | --- | --- | --- |
| C | **libbpf** | Kernel 团队 | 官方、稳定、生产首选 |
| C++ | libbpf + RAII 包装 | 各团队自写 | 没有标准 wrapper；boost::asio 风格的 safe wrapper 各家实现 |
| Rust | **aya** | aya-rs 社区 | 纯 Rust，无 libbpf 依赖，单 binary 部署最友好 |
| Rust | libbpf-rs + libbpf-sys | community | 包 libbpf 的 FFI，体积小但要动态链接 libbpf.so |
| Rust | redbpf | Datadog（已不活跃） | 早期 Rust eBPF，停留在旧内核 |
| Go | **cilium/ebpf** | Isovalent / Cilium | 纯 Go，无 cgo，部署极简 |
| Go | ebpf-go（早期） | community | 已合并进 cilium/ebpf |
| Python | **BCC** | IO Visor 社区 | 运行时 fork clang 编译，启动慢 |
| Python | bpftrace 的 Python 绑定 | bpftrace 项目 | 用 bpftrace 后端写 Python 脚本 |
| Lua | BCC Lua bindings | IO Visor | 已不活跃 |
| Java | **bpf-pulse / jbpf** | Intel + community | JVM 上跑 BPF，main mode + interp mode |
| .NET | ebpf-for-windows 的托管绑定 | Microsoft | 跨平台（Windows 也用），份额小 |

### 选用速记

* **C 项目**：libbpf + skeleton，无脑选
* **Rust 服务**：aya，单 binary 无动态依赖，CD / 镜像最小
* **Go 服务**：cilium/ebpf，避免 cgo 链
* **临时排查**：bpftrace / BCC Python
* **Java 系**：jbpf，但生态尚浅

## ④ 高层产品

按场景分组：

### 网络

* **Cilium**（Isovalent）：CNCF 毕业，eBPF datapath CNI，替代 kube-proxy iptables，Hubble 做可观测，Tetragon 做安全。一套吃三家
* **Calico eBPF dataplane**（Tigera）：另一种 CNI 实现，纯 eBPf 模式做路由 / policy
* **Katran**（Meta 开源）：L4 LB，Cloudflare 也在用
* **Cilium Service Mesh**：Istio ambient 的底层，eBPF 做 L4 + sidecar 卸载 L7
* **bpf-proxy**：cilium 团队做的纯 eBPF L4 proxy

### 安全

* **Falco**（CNCF incubating）：runtime 安全检测，eBPF 抓 syscall，rules 引擎告警
* **Tetragon**（Isovalent / Cilium 旗下）：直接 eBPF，kernel-level policy enforcement，比 Falco 更低层更高效
* **Tracee**（Aqua Security）：runtime 安全 + forensics
* **Landlock**（内核自带）：LSM 框架，让进程声明自己的访问控制策略
* **bpfd**（Red Hat）：特权 daemon，统一管理 BPF prog / map，普通进程通过 RPC 用

### 可观测

* **Pixie**（New Relic）：K8s 应用层观测，自动 instrument，eBPF + eBPF map + 内置查询 DSL（Pixielang）
* **Hubble**（Isovalent）：Cilium 的网络可观测，service map / flow log
* **Parca**（Polar Signals）：continuous profiling，eBPF 做 on-CPU / off-CPU profile，类似 Google-Wide Profiling
* **bpftrace / BCC tools** 集：`biolatency`, `capable`, `tcpconnect`, `opensnoop`, `execsnoop`, `biotop` ... 内核工具合集
* **DeepFlow**（云杉网络）：国产 APM，eBPF 做 auto-instrumentation
* **Kindling**（观测云）：eBPF APM

### 调度 / 存储 / 其他

* **sched_ext**（Linux 6.12+）：BPF 实现调度器，Meta 的 `scx_*` 调度器（`scx_layered`, `scx_rusty`, `scx_bpfland`）已经合入
* **io_uring + eBPF**：io_uring 的 opcode / latency profile
* **btrfs / bcachefs** 的 eBPF hook：文件系统层 trace

## ⑥ 学习路径

按难易度：

1. **bpftrace 一行命令** → 解决手头 80% 的"看一眼"需求
2. **BCC Python 工具** → 把现成 tool 改一改
3. **libbpf C + 一两个简单 prog** → 理解 CO-RE、ringbuf、verifier
4. **aya Rust** → 写生产级 eBPF 服务
5. **cilium/ebpf** → Go 系最佳实践
6. **调 verifier / 改内核 BPF 子系统** → C + 内核阅读

参考资料：

* **官方**：<https://docs.kernel.org/bpf/>，libbpf README，bpftool man page
* **书籍**：《Learning eBPF》（Liz Rice，O'Reilly 2023）— 入门首选
* **博客**：Andrii Nakryiko 的 libbpf + CO-RE 解析，Brendan Gregg 的 BPF performance tools 系列
* **课程**：Isovalent 的 Cilium eBPF 培训（前身 Cilium 官方课）
* **代码库**：<https://github.com/libbpf/libbpf-bootstrap>（脚手架），<https://github.com/cilium/ebpf>（Go），<https://github.com/aya-rs/aya>（Rust）

## ⑦ 选型一句话

* **排查 / 临时**：bpftrace
* **生产用户态服务**：C → libbpf，Rust → aya，Go → cilium/ebpf
* **K8s 网络 / 观测 / 安全**：Cilium 一套（Hubble + Tetragon）
* **Runtime 安全**：Falco（rules 引擎丰富） / Tetragon（性能 + 内核拦截）
* **Continuous profiling**：Parca / Pixie
* **调度研究**：sched_ext + scx_*
* **不熟悉任何底层**：先用 Pixie / Hubble 看现成数据，再决定要不要自写

## ⑧ 当前趋势（2025–2026）

* **sched_ext** 成熟，多家公司上线生产（Meta 主力）
* **BTF-only 内核**讨论：未来可能去 vmlinux 文本符号表，全靠 BTF
* **Cilium + Tetragon** 合并到 CNCF 主流，K8s 网络事实标准之一
* **Rust aya** 在新项目里份额持续涨
* **bpftrace** 仍稳坐脚本之王，单行排查无替代
* **eBPF for Windows** 持续推进，跨平台 BPF 工具链成为可能
* **AI 辅助 BPF 开发**：用 LLM 写 bpftrace / aya 程序降低入门门槛（主要是调试场景）
