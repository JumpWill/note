## 设计原则

K8s 的扩展性全部建立在 **"所有东西都是 API"** 上：

* 组件之间走 gRPC + protobuf（性能、跨语言、强类型）
* 资源对象用 OpenAPI / CRD 描述（声明式）
* 控制面靠 **controller loop**（调谐）+ **webhook**（拦截）
* 接入插件都按 **"实现一个 gRPC server + 注册到 kubelet / kube-apiserver"** 这个套路

整套插件接口分类：

```text
┌─ 控制面插件 ──────────────────────────────────────┐
│  CNI（网络）    CRI（容器运行时）  CSI（存储）     │
│  Device Plugin（GPU/FPGA 等硬件）                  │
│  Scheduler Extender（自定义调度）                  │
│  Aggregated APIServer（嵌套 API）                  │
└────────────────────────────────────────────────────┘
┌─ 资源 / 行为扩展 ─────────────────────────────────┐
│  CRD（自定义资源）                                 │
│  Admission Webhook（Mutating / Validating）       │
│  RuntimeClass（运行时分类）                        │
└────────────────────────────────────────────────────┘
┌─ 认证 / 授权 ─────────────────────────────────────┐
│  AuthN：Token / Webhook / OIDC / Client Cert      │
│  AuthZ：RBAC / Node / Webhook / ABAC（弃用）      │
└────────────────────────────────────────────────────┘
┌─ 监控 / 集成 ─────────────────────────────────────┐
│  Metrics：cAdvisor / kube-state-metrics / Custom  │
│  Cloud Provider（云厂商适配）                      │
│  Network Service Mesh / SMI（服务网格）            │
└────────────────────────────────────────────────────┘
```

## 三大基础接口

### CNI（Container Network Interface）

* 用途：**Pod 网络**，IP 分配 + 路由 + 网络策略
* 实现位置：`/opt/cni/bin/<plugin>` 二进制，由 kubelet 通过 stdin JSON 调用
* 协议：[cni.dev](https://www.cni.dev/) 规范，主流程 `ADD / DEL / CHECK / VERSION / GC`
* 代表实现：Calico、Cilium、Flannel、Multus（多网卡）、Weave
* 关键概念：IPAM（dhcp / host-local / whereabouts）、Chained Plugins（calico + portmap）

→ 详见 [CNI/](./CNI/)

### CRI（Container Runtime Interface）

* 用途：**容器运行时**，kubelet 不直接调 docker/containerd，而是走 CRI
* 协议：gRPC，protobuf
* 关键 service：`RuntimeService`（PodSandbox / Container / Exec / PortForward / Logs）、`ImageService`
* 代表实现：containerd（`cri` plugin）、CRI-O、Frakti（已不维护）
* 配套规范：OCI Runtime Spec（runC / crun / gVisor / kata-containers 都符合）、OCI Image Spec

→ 详见 [CRI/](./CRI/)

### CSI（Container Storage Interface）

* 用途：**块 / 文件存储**，让外部存储厂商按统一接口接入
* 协议：gRPC，`Identity / Controller / Node` 三个 service
* 关键能力：Volume Create/Delete、Attach/Detach、NodePublish/Unpublish（mount 到 Pod）、Expand、Snapshot
* 代表实现：Ceph RBD / CephFS、AWS EBS / EFS、GCP PD、Azure Disk / File、阿里云盘、Longhorn、OpenEBS、JuiceFS
* 关键概念：External Provisioner / External Attacher / External Resizer / External Snapshotter — 都是 **sidecar controller**，跟 CSI driver 配对跑

→ 详见 [CSI/](./CSI/)

## 控制面扩展

### Device Plugin

* 用途：让 kubelet 识别 **非标准硬件**（GPU、FPGA、TPU、RDMA、SR-IOV VF、高性能网卡等）
* 协议：gRPC，kubelet ↔ Device Plugin 之间，DevicePlugin service：`ListAndWatch / Allocate / GetDevicePluginOptions / GetPreferredAllocation / PreStartContainer`
* 注册路径：`/var/lib/kubelet/device-plugins/<name>.sock` + `Registration` RPC
* 代表实现：NVIDIA device-plugin for GPU、AMD GPU、Intel QAT、SRIOV
* 配套：**DRA（Dynamic Resource Allocation）**（1.26+ GA，1.34 默认）是 Device Plugin 的下一代，用 ResourceClaim + ResourceClaimTemplate，更灵活

### Scheduler Extender

* 用途：自定义调度决策，把 kube-scheduler 不擅长的事交给外部
* 协议：HTTP，POST 一个 JSON：节点列表 + pod spec → extender 返回 `hostnames[]`（可分数）or 修改过 spec
* Hook 点：`filter / score / pre-bind / permit / reserve / pre-provision`
* 注意：性能差，**v1.32+ 推荐用 Scheduler Framework + Plugin API**（Go in-tree 插件），自定义逻辑直接跑在 scheduler 进程里，延迟从 ms 降到 μs
* 现状：Scheduler Extender 是 **过渡方案**，新项目直接上 Scheduler Plugin

### Aggregated APIServer（AA）

* 用途：在 kube-apiserver 后面挂一个 **独立的 APIServer**，对外看起来是同一个 apiserver（`/apis/xxx.example.com/v1/...`）
* 协议：标准 Kubernetes API（CRD 那套）
* 代表：`metrics-server`（HPA 的数据源）、`k8s.cni.cncf.io/v1`、各种 Operator 自带的 APIServer
* 配套：APIService 对象声明路由 + CA 证书链

## 资源 / 行为扩展

### CRD（CustomResourceDefinition）

* 用途：声明自定义资源，让 `kubectl apply -f mything.yaml` 能管你的自定义对象
* 协议：OpenAPI v3 schema
* 配套：**Controller** 用 controller-runtime（kubebuilder / operator-sdk 生成），调谐 CRD 实例
* 进阶：`conversion` webhook 支持多版本、struct / list 类型验证
* 几乎所有 Operator 模式（etcd-operator、cert-manager、ArgoCD、Istio）都基于 CRD

### Admission Webhook

* 用途：在对象持久化到 etcd **前** 拦截修改 / 校验
* 两种：
  * **Mutating**：改 spec（注入 sidecar、注入 toleration）
  * **Validating**：拒绝不合规请求
* 协议：HTTPS，AdmissionReview API
* 顺序：Mutating → Validating → 持久化
* 代表：Istio sidecar 注入、cert-manager、Kyverno / OPA Gatekeeper、PodSecurityPolicy（已弃，被 PSA 替代）

### RuntimeClass

* 用途：选择 Pod 跑的运行时（runC / gVisor / kata / Windows containers）
* 配置：在 `RuntimeClass` 对象里声明 handler，由 kubelet 通过 CRI 把 RuntimeHandler 传给 runtime
* 典型用法：untrusted workload 用 gVisor / kata 强隔离

## 认证 / 授权

### AuthN（认证）

* 标识请求方是谁，输出 `user + group + extra`
* 实现路径（按优先级）：
  * **Client cert**（CN=`<user>`、O=`<group>`）
  * **Bearer Token**（ServiceAccount JWT，签到 kube-apiserver 内置 OIDC）
  * **Bootstrap Token**（kubeadm join 用）
  * **Webhook Token Authentication**：自建 OIDC 服务 / LDAP，对接到 apiserver

### AuthZ（授权）

* 回答"这个 user 能对这个 resource 做这个 verb 吗"
* 内置：
  * **Node**：kubelet 专用，只允许读自己节点的 Pod
  * **RBAC**：最常用，Role / ClusterRole + RoleBinding / ClusterRoleBinding
  * **Webhook**：自定义授权（外部 OPA / SpiceDB / 业务策略）
* **ABAC** 已弃用，别用

### 其他

* **ServiceAccount**：Pod 内的身份，token 通过 projected volume 挂进 `/var/run/secrets/kubernetes.io/serviceaccount/`
* **PodSecurity Admission**（替代 PSP）：namespace 标签 `pod-security.kubernetes.io/enforce=baseline|restricted|privileged`

## 监控 / 集成

### Metrics

* `/metrics` 端点，Prometheus 格式，**注意是 OpenMetrics 不是 OpenTelemetry**
* 三层：
  * **kubelet / 容器**：`cadvisor` 内置在 kubelet，CPU / mem / net / fs
  * **kube-state-metrics**：K8s 对象级别（Deployment 副本数 / Pod status）
  * **Custom**：应用自己暴露 `/metrics`
* `metrics-server`（AA 部署）给 HPA / VPA / `kubectl top` 提供内存 API

### Cloud Provider

* 用途：让 kube-controller-manager 能跟云厂商 API 通信（LoadBalancer、Route、PersistentVolume 回收）
* 模式：in-tree（v1.28 起大部分已移出）→ **cloud-controller-manager**（独立组件）+ **Out-of-Tree Provider**（阿里云 / AWS / Azure / GCP 都提供）
* 关联：LoadBalancer Service → 云 SLB；Node lifecycle → 云 ASG 扩缩

### Service Mesh 接口（SMI / GAMMA）

* 用途：让 Istio / Linkerd / Consul / Kuma 走同一套 mesh 抽象
* **SMI**（Service Mesh Interface，CNCF）：TrafficSplit / TrafficAccess / HTTPRouteGroup 等 CRD
* **GAMMA**（Gateway API for Mesh）：CNCF Gateway API 的 mesh 扩展，正在替代 SMI
* 实操：今天**优先 Gateway API**（含 GAMMA），SMI 已基本冻结

## 一句话选型

* 给 Pod 配网 → **CNI**
* 接新容器运行时 → **CRI**
* 接存储 → **CSI**
* 暴露 GPU / 特殊硬件 → **Device Plugin** 或更新的 **DRA**
* 自定义资源 + 控制器 → **CRD + controller-runtime**
* 在 apiserver 落库前改 / 校验对象 → **Admission Webhook**
* 嵌套 API 服务 → **Aggregated APIServer**
* 自定义调度决策 → 老方案 **Scheduler Extender**，新方案 **Scheduler Plugin**
* 自定义认证 → **Webhook Token Authn**
* 自定义授权 → **RBAC** 配 **Webhook Authorization**

## 学习路径

1. 把 `kubectl describe <pod>` 看懂，能在 events 找到调度 / mount / 网络问题
2. 自己写一个 CNI plugin demo（最简单，200 行 Bash 就能跑）
3. 自己写一个简单的 CSI driver（`local-csi-driver` 入门）
4. 写一个 CRD + controller（`kubebuilder init` 起手）
5. 写一个 Admission Webhook（`kubewebhook` / `cert-manager`）
6. 折腾 RuntimeClass + gVisor / kata
7. 看 Aggregated APIServer（metrics-server 源码）
